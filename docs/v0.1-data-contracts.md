# DragonBoat v0.1 Data Contracts

DragonBoat v0.1 uses local, append-only data contracts.

The goal is to make each run observable and replayable without binding the core product to any single coding-agent CLI.

## Storage Layout

The canonical v0.1 runtime layout should be:

```text
.dragonboat/
  runs/
    run_<id>/
      events.ndjson
      state.json
      crew.json
      tasks/
        task_<id>/
          task-packet.json
          task-packet.md
      evidence/
        task_<id>/
          evidence-bundle.json
          summary.md
      mailbox/
        inbox/
          agent_<id>.ndjson
        outbox/
          agent_<id>.ndjson
      logs/
        cmd_<id>.log
      artifacts/
```

`events.ndjson` is canonical. Other files are materialized artifacts for agent readability, validation, desktop rendering, and replay.

## Local Event

Schema: [`schemas/v0/local-event.schema.json`](../schemas/v0/local-event.schema.json)

Events are append-only JSON objects, one per line.

Each event contains:

- stable schema version
- event id
- run id
- monotonically increasing sequence number
- timestamp
- event type
- actor
- optional task, message, or command reference
- event-specific payload

The command deck should render state from the event stream rather than trusting a mutable UI state file.

## Event Types

v0.1 event types are:

- `run.created`
- `run.completed`
- `run.failed`
- `crew.member.registered`
- `crew.member.status_changed`
- `task.packet.created`
- `task.plan.submitted`
- `task.plan.approved`
- `task.plan.rejected`
- `task.status_changed`
- `mailbox.message.sent`
- `mailbox.message.delivered`
- `mailbox.message.acknowledged`
- `mailbox.message.failed`
- `command.started`
- `command.output`
- `command.finished`
- `evidence.submitted`
- `steerer.review.completed`

The desktop can derive crew state, task state, mailbox timelines, and evidence panels from these events.

## Task Packet

Schema: [`schemas/v0/task-packet.schema.json`](../schemas/v0/task-packet.schema.json)

A task packet is the handoff from steerer to rower.

The JSON version is canonical. The Markdown version is generated from the JSON packet so CLI agents can read it naturally.

Every task packet includes:

- task id
- run id
- title
- assigned rower
- creating steerer
- objective
- relevant context
- worktree and path boundary
- dependencies
- acceptance criteria
- expected evidence
- reporting requirements

The packet should be strict enough for automation and readable enough for agents.

## Evidence Bundle

Schema: [`schemas/v0/evidence-bundle.schema.json`](../schemas/v0/evidence-bundle.schema.json)

An evidence bundle is the rower's completion package.

It must include:

- task id
- run id
- submitting agent
- success, failure, blocked, or partial status
- summary
- changed files
- commands run
- check results
- mailbox references
- artifacts
- risks
- unresolved questions
- next-step recommendations

The steerer should review evidence bundles before accepting or integrating any work.

## Mailbox Model

Mailbox messages are represented as events and optionally materialized to per-agent inbox/outbox files.

The event log remains the source of truth.

Message types should map to event payloads:

- `question`
- `contract`
- `status`
- `blocker`
- `review`
- `evidence`

Each message should include:

- sender
- recipient
- related task id
- message type
- body
- delivery state

For v0.1, delivery can be soft real-time. Hooks, MCP tools, or plugins can write events immediately. Agents without push support can poll their inbox file.

## ID Conventions

Use stable, readable prefixes:

- `run_`
- `agent_`
- `task_`
- `msg_`
- `cmd_`
- `evt_`
- `evidence_`

IDs should be unique within the local run store and safe for filenames.

## Design Rule

DragonBoat should never require a worker to claim "done" without evidence.

Evidence is the product boundary between worker execution and steerer trust.
