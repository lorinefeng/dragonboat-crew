# DragonBoat v0.1 Technical Decisions

This document records the initial implementation baseline after the product direction was approved.

These decisions are scoped to the v0.1 demo and can be revised after real runs with Codex and Claude Code.

## Decision 1: Canonical Runtime State

Use local files under `.dragonboat/runs/<run_id>/`.

The canonical source of truth is an append-only `events.ndjson` file. Task packets and evidence bundles are stored as validated JSON artifacts with generated Markdown companions for agent readability.

Why:

- local-first
- easy to inspect
- easy to replay
- easy for agents to read
- simple enough for v0.1

Rejected for v0.1:

- database-first storage
- cloud event service
- opaque binary logs

## Decision 2: Schema Format

Use JSON Schema Draft 2020-12 for v0.1 contracts.

The initial schemas are:

- [`schemas/v0/local-event.schema.json`](../schemas/v0/local-event.schema.json)
- [`schemas/v0/task-packet.schema.json`](../schemas/v0/task-packet.schema.json)
- [`schemas/v0/evidence-bundle.schema.json`](../schemas/v0/evidence-bundle.schema.json)

Why:

- supported by both human docs and agent outputs
- compatible with structured-output validation patterns
- language-neutral
- easy to version

## Decision 3: Demo App Stack

Use a small TypeScript full-stack app:

- frontend: Vite + React + TypeScript
- backend: Hono on Node.js
- tests: Vitest for unit/integration checks, Playwright for end-to-end demo checks
- storage: in-memory or local JSON file for the demo app only

Why:

- frontend, backend, and QA/Ops work can be split naturally
- TypeScript keeps contracts readable for agents
- Vite has a fast local developer loop and official React TypeScript templates
- Hono is small, TypeScript-first, and built on Web Standards
- Playwright gives a clear QA/Ops story for visible web flows

Rejected for the first demo:

- Next.js, because it collapses too much frontend/backend work into one framework for the coordination demo
- a database, because it adds operational noise before DragonBoat has proven its coordination loop
- a production SaaS example, because the launch demo should emphasize the crew workflow rather than product complexity

## Decision 4: Desktop Stack

Use Tauri 2 + React + TypeScript for the command deck.

Why:

- local-first desktop identity
- small app footprint
- native file watching and command integration through a Rust backend
- WebView frontend is enough for the command deck and replay experience
- the frontend stack can share visualization components with future web demos

Electron remains a fallback if Tauri blocks launch speed, but it should not be the first choice because it ships Chromium and Node.js inside the app.

## Decision 5: Agent Integration Strategy

Start with CLI adapters.

The v0.1 adapter order should be:

1. file-based contracts and event log
2. Codex CLI steerer adapter
3. Claude Code CLI rower adapter
4. file-polling mailbox
5. MCP mailbox tools
6. richer hooks/plugins after the first demo

Why:

- file contracts are debuggable
- CLI adapters prove the actual user workflow
- soft real-time is enough for v0.1
- MCP should improve the interface without becoming the core product

## Decision 6: Worktree Strategy

Require one isolated git worktree per rower in v0.1.

Why:

- prevents conflicting edits during the first demo
- makes evidence easier to inspect
- makes rollback safer
- gives each rower a clean identity and local workspace

Shared-worktree parallel editing is a future goal, not a v0.1 requirement.

- Hono is a small TypeScript web framework built on Web Standards: https://www.honojs.com/docs/
- Playwright provides end-to-end testing for modern web apps: https://playwright.dev/docs/intro
